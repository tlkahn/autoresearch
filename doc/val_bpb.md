`val_bpb`（validation bits per byte）是一种衡量语言模型预测能力的评估指标，它表示模型平均需要多少比特来编码验证集中的每个字节——数值越低，说明模型越好。

## 一、什么是 bits per byte？

要理解 `val_bpb`，先要理解信息论中的一个基本概念：**编码**。

假设你和朋友玩猜词游戏。朋友脑中想了一个字，你来猜。如果你猜得很准（比如你说「是不是'的'？」，对方说「对！」），你只需要很少的信息来确认答案。如果你猜得很差（比如对方想的是一个生僻字），你就需要反复提问，消耗大量信息。

**Bits per byte（BPB）**衡量的正是这种「猜测效率」：模型平均需要多少个比特（bit）来编码输入文本的每个字节（byte）。

下面是一个直觉上的对照。

```
val_bpb ≈ 1.0   →  模型每个字节只需 1 比特即可表达，压缩效率很高
val_bpb ≈ 8.0   →  模型完全无法压缩，相当于随机猜测（1 字节 = 8 比特）
val_bpb ≈ 0.5   →  接近人类级别的预测能力（理论下限取决于语言本身的熵）
```

## 二、BPB 和 Perplexity 有什么区别？

在语言模型评估中，另一个常见的指标是 **Perplexity（PPL，困惑度）**。两者都在衡量模型的预测能力，但有一个关键区别。

**Perplexity** 依赖分词器（tokenizer）。不同的分词器把文本切成不同的 token 序列——GPT 系列使用 BPE，有些模型使用字符级分词，有些使用词级分词。同一段文本，在不同分词方案下，token 数量不同，计算出的 perplexity 也不同。这意味着使用不同分词器的模型之间，perplexity 无法直接比较。

**BPB** 则绕过了这个问题。它把所有东西归一化到字节级别——无论你的分词器把文本切成多少个 token，最终都换算成「每个字节需要多少比特」。

用一个类比来说明：perplexity 就像用「每公里油耗」来比较两辆车的燃油效率，但一辆车的里程表是英里制、另一辆是公里制——数字没法直接比。BPB 相当于统一换算成「每公里消耗多少升汽油」，不管仪表盘显示的是什么单位。

两者的数学关系如下。

```
BPB = cross_entropy_loss / ln(2) / bytes_per_token
```

上面公式中，`cross_entropy_loss` 是模型在 token 级别的交叉熵损失，`ln(2)` 用于将自然对数（nats）转换为比特（bits），`bytes_per_token` 是每个 token 平均对应多少字节（取决于分词器和文本内容）。

## 三、val_bpb 是最好的评估指标吗？

不是。val_bpb 是一个好指标，但不是万能的。

它擅长衡量的是模型的**原始预测能力**——在统计意义上，模型对文本的「理解」有多深。但现实中，我们对语言模型的期望远不止于此。

下面是一张对照表，展示了不同场景下应该用什么指标。

```
┌──────────────────────┬────────────────────────────────────┐
│ 你想衡量什么          │ 应该用什么指标                       │
├──────────────────────┼────────────────────────────────────┤
│ 原始语言建模能力       │ BPB、Perplexity、交叉熵              │
│ 知识与推理能力         │ MMLU、ARC、HellaSwag、GSM8K          │
│ 代码生成能力          │ HumanEval、MBPP                     │
│ 指令遵循能力          │ MT-Bench、AlpacaEval                │
│ 安全性与可靠性        │ TruthfulQA、毒性检测基准              │
│ 多语言能力           │ MMLU-ProX（覆盖 29 种语言）           │
└──────────────────────┴────────────────────────────────────┘
```

### 3.1 BPB 捕捉不到的东西

（1）**下游任务表现**——val_bpb 更低的模型不一定在问答、摘要或翻译任务上表现更好。两者之间存在正相关，但不是必然关系，尤其是当 bpb 差距很小的时候（比如 0.997 vs 1.000）。

（2）**推理与事实准确性**——val_bpb 只衡量「预测下一个 token 的概率分布有多准」，不衡量模型是否真正「理解」了内容。一个模型可能在统计上拟合得很好，但在逻辑推理上一塌糊涂。

（3）**安全性与对齐**——bpb 完全不关心模型是否会产生有害、偏见或幻觉内容。一个 bpb 极低但会产生危险输出的模型，在实际部署中毫无价值。

（4）**效率**——bpb 忽略了推理成本。一个参数量多 10 倍、bpb 只好 0.001 的模型，在工程实践中可能是更差的选择。

（5）**校准度（Calibration）**——模型可能在某些 token 上极度自信但完全错误，而 bpb 会把这些错误「平均」掉，掩盖了局部的灾难性失败。

### 3.2 为什么 autoresearch 选择 BPB？

尽管 BPB 不是万能的，但它是这个项目的最佳选择。原因如下。

（1）**任务是纯预训练**——没有微调（fine-tuning）或人类反馈强化学习（RLHF），不涉及下游任务。在预训练阶段，BPB 是最直接的评估手段。

（2）**智能体可能改变架构**——如果换了模型结构，词汇表大小可能变化，此时需要一个不依赖分词器的指标。BPB 的字节级归一化完美满足这个需求。

（3）**单一数值，便于机械比较**——自动化循环需要一个明确的判断标准：这次实验比上次好还是差？BPB 就是一个数字，低了就保留，高了就回滚。不需要人类介入判断。

（4）**模型太小，下游基准无意义**——5 分钟训练出来的是一个约 50M 参数的小模型。让这样的模型去做 MMLU（大学考试级别的多项选择题）或 HumanEval（代码生成），结果基本就是随机猜——毫无参考价值。

## 四、主流的 LLM 评估基准

为了更完整地理解 val_bpb 的定位，下面简要介绍几个主流的评估基准。

### 4.1 MMLU

**MMLU**（Massive Multitask Language Understanding）包含 57 个学科的 15,908 道选择题，涵盖数学、历史、法律、伦理等领域。它是目前评估 LLM 通用知识最广泛使用的基准。截至 2026 年初，顶级模型在 MMLU-Pro（加强版）上的准确率已接近 90%。

### 4.2 HumanEval

**HumanEval** 用于评估代码生成能力。它包含 164 个编程问题，模型需要生成能通过单元测试的 Python 函数。

### 4.3 GSM8K

**GSM8K** 包含 8,500 道小学数学应用题。它测试的不是高等数学知识，而是多步推理能力——模型能否正确地把一个问题拆解成若干步骤并逐步计算。

### 4.4 TruthfulQA

**TruthfulQA** 测试模型是否会生成人类常见的错误信息。例如：「人只使用了大脑的 10% 吗？」——正确答案是否定的，但许多模型会给出肯定的错误回答。

```

┌───────────┐      ┌───────────────┐     ┌──────────────┐

│           │      │ MT-Bench      │     │ 用户满意度     │
└───────────┘      └───────────────┘     └──────────────┘
      ▲                                        ▲
      │                                        │
  autoresearch                          实际产品需要关心的
  只关心这里
```

val_bpb 不是「最好的」评估指标——不存在这样一个指标。但对于 autoresearch 这个特定项目——纯预训练、自动化循环、小模型——它是最合适的选择。
